# Getting Support for H0NEYP0T-466/Dr.Document

Welcome to the Dr. Document project! We're excited to have you here. This guide will help you get the support you need for this AI-powered GitHub documentation generator.

## 🚀 Support Channels

We offer several ways to get help with Dr. Document:

- **GitHub Issues**: For bug reports, feature requests, and technical problems
- **GitHub Discussions**: For questions, community help, and general conversation
- **Documentation**: Comprehensive guides and API references

## 🐛 GitHub Issues

Use GitHub Issues for:

- **Bug reports** - App crashes, unexpected behavior, or incorrect documentation generation
- **Feature requests** - New agent capabilities or UI improvements
- **Integration problems** - Issues with OpenAI API, GitHub API, or WebSocket connections
- **Configuration problems** - Environment setup, dependency issues, or deployment problems

### What to include in your issue:

✅ **Required information:**
- Dr. Document version (check `package.json` and backend requirements)
- Python version and Node.js version
- Operating system and architecture
- Relevant error messages (from logs, browser console, or terminal)
- Steps to reproduce the issue
- Sample repository URL (if applicable)

✅ **Helpful additions:**
- Screenshots of the problem
- Relevant log output (from `backend/logger.py`)
- Your environment variables (redact sensitive info like API keys)
- Expected vs. actual behavior

❌ **Don't create issues for:**
- General usage questions
- Setup help
- Feature discussions
- Security vulnerabilities (see below)

## 💬 GitHub Discussions

Use GitHub Discussions for:

- **General questions** about how to use Dr. Document
- **Setup and configuration help**
- **Best practices** for documentation generation
- **Feature discussions** and brainstorming
- **Community help** and knowledge sharing
- **Show and tell** - Share your generated documentation!

Choose the appropriate category:
- **Q&A** - For specific questions
- **Ideas** - For feature suggestions and improvements
- **Show and tell** - For sharing results and experiences
- **General** - For other community conversations

## 🤔 How to Ask a Good Question

### For Issues:
1. **Search first** - Check existing issues and discussions
2. **Be specific** - Include exact error messages and reproduction steps
3. **Provide context** - Explain what you're trying to achieve
4. **Include minimal example** - Create a small test case if possible
5. **Check your setup** - Verify you're using compatible versions

### For Discussions:
1. **Choose the right category** - Use Q&A for questions, Ideas for suggestions
2. **Provide background** - Explain your use case and goals
3. **Be clear and concise** - Use proper formatting for code and commands
4. **Include relevant details** - OS, versions, configuration snippets

## ⏰ Response Time Expectations

We aim to provide timely responses:

- **Issues**: 2-5 business days for initial response
- **Discussions**: 3-7 business days for community engagement
- **Critical bugs**: Priority review within 24-48 hours
- **Security issues**: Immediate attention (see below)

Response times may vary during:
- Major releases
- Holiday periods
- Community events

## 🔒 Security Issues

**Do NOT create public issues for security vulnerabilities.**

For security concerns:
1. **Email the maintainer directly** (check repository settings)
2. **Include detailed reproduction steps**
3. **Provide potential impact assessment**
4. **Do not share sensitive data in public channels**

We follow responsible disclosure practices and will work with you to resolve security issues promptly.

## 🚫 What NOT to Do

❌ Don't create duplicate issues - search first
❌ Don't include sensitive information (API keys, passwords)
❌ Don't spam issues or discussions
❌ Don't create issues for general questions (use Discussions)
❌ Don't expect immediate responses to every message
